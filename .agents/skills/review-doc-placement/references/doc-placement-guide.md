# Document Placement Guide

## 目的

- このガイドは、記述の種類ごとに適切な配置先を判断するために使う。

## 文書ごとの責務

### `AGENTS.md`

- コーディングエージェントの作業ルール、レビュー手順、完了条件を書く。
- 文書更新の運用ルールや、どのスキルを使うかの指示を書く。
- 実装規約やユーザー向け仕様を詳しく書かない。

### `docs/coding-standards.md`

- コードレビューで違反判定できる実装規約を書く。
- 言語、レイヤ、ライブラリごとの実装上の制約を書く。
- 作業手順、レビュー運用、仕様説明、ユーザー向け操作方法を書かない。

### `docs/planty-wiki specification.md`

- ユーザーに見える挙動、サポート範囲、機能制約、非機能要件を書く。
- 実装手段ではなく、満たすべき振る舞いを書く。
- 特定API名、内部構造名、実装テクニックへの依存は避ける。

### `docs/planty-wiki markdown syntax.md`

- ユーザーが入力できるMarkdown記法と、その具体例を書く。
- 対応している構文と非対応構文を書く。
- 文書運用ルールや実装上の事情を書かない。

### ルートの `README.md`

- プロジェクト全体の概要、主要機能、使い始め方を書く。
- 外部の利用者や開発者が最初に参照する入口として使う。
- 詳細な仕様本文や局所的な実装規約は抱え込まない。

### 配下ディレクトリの `README.md`

- ディレクトリ内にあるコードや文書への導線を書く。
- 詳細ルールや仕様本文を書かない。

### `docs/AGENTS.md`

- `docs/` 配下の文書作成ルールを書く。
- 文体、例の付け方、表記の統一など、文書運用に閉じた指示を書く。
- アプリ全体の実装規約や仕様本文を書かない。

### 配下ディレクトリの `AGENTS.md`

- そのディレクトリ配下にだけ適用するルールを書く。
- たとえば `.github/workflows/AGENTS.md` には、ワークフロー実装に限定したルールを書く。
- リポジトリ全体に効くルールはルートの `AGENTS.md` に集約する。

## 判断ルール

- エージェントの行動を指示する記述は `AGENTS.md` に置く。
- レビューで違反判定する実装ルールは `docs/coding-standards.md` に置く。
- ユーザーが期待する振る舞いは仕様書に置く。
- ユーザーが実際に書く記法はMarkdown構文書に置く。
- 文書の入口や参照順だけを示す内容は `README` に置く。
- 特定ディレクトリだけに適用する文書ルールは、そのディレクトリの `AGENTS.md` に置く。

## 典型的な配置ミス

- 実装テクニックが仕様書に入っている。
- 文書更新ルールが構文書や仕様書に入っている。
- ユーザー向け記法の説明が `AGENTS.md` や規約書に入っている。
- プロジェクト全体のルールが配下ディレクトリの `AGENTS.md` に入っている。
- 同じ主題の記述が、仕様書と規約書の両方に重複している。

## 報告の仕方

- 現在の配置先と、移すべき配置先を明記する。
- その記述が「運用ルール」「実装規約」「仕様」「構文説明」「導線」のどれかを示す。
- 移動先で重複や矛盾が起きる場合は、その相手文書もあわせて指摘する。
